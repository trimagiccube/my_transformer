Big Picture

一次前向 = 把一个 token 变成 32000 个 logit

推理是自回归的:每次只喂入 1 个 token,经过 6 个解码层算出词表上每个词的分数(logits),采样得到下一个 token,再喂回去。下面是 forward()(run.cu:623)的完整骨架——注意它只有 6 类 GPU 操作反复组合。

计算的 6 块积木

  • RMSNorm — 归一化,自写 kernel(CUB 归约)
  • MatMul — 所有线性层,交给 cuBLAS
  • RoPE — 旋转位置编码,自写 kernel
  • Attention — 打分+softmax+加权,自写 kernel,读写 KV Cache
  • SwiGLU — 前馈激活,自写 kernel
  • Accum — 残差相加,自写 kernel

数据在哪

  • 权重:启动时 mmap + 一次性 cudaMemcpy 全部搬上显存,常驻 GPU
  • 激活值 (x, xb, q, k, v, att, hb...):全部在显存,层与层之间不回主机
  • KV Cache:显存里一块 [layer][seq][kv_dim] 大数组,每步写入当前位置
  • 每步唯一的主机↔设备拷贝:开头拷 1 行 embedding 上去,结尾把 logits 拷回来采样

CUDA Execution Model

"丢给 GPU 算" 到底是什么意思

CPU 端代码长这样:kernel<<<grid, block>>>(args)。三个尖括号告诉 GPU 启动多少个并行线程,以及怎么分组。理解这个就理解了全部"调度"。

两个常数决定一切

run.cu 全局只用两种线程块大小:

const int num_threads_lrg = 1024; // 大
const int num_threads_med = 256; // 中
  • grid = 启动多少个 线程块 (block)。常按"输出有多少个独立单元"来定,比如注意力按 n_heads 个头分块。
  • block = 每块多少 线程 (thread)。块内线程能共享内存 + 同步,用来做归约(求和/求最大)。

两种并行模式

  • 逐元素型(silu / accum):每个线程管一个元素,grid=divUp(N,256),无需通信。简单粗暴。
  • 归约型(rmsnorm / softmax / attention):一个 block 内 1024 个线程协作处理一个向量/一个头,用 cub::BlockReduce 把 1024 个部分和合成一个总和。
  • 矩阵乘:不自己写,调 cublasSgemv。因为 GEMM 想写快极难,库已是性能天花板。

例:多头注意力的线程分工 multi_head_attention_kernel<<<6, 1024>>>

启动 6 个 block(每个头一块),每块 1024 个线程协作:

谁干什么怎么并行
blockIdx.x (0~5)负责第 h 个注意力头6 个头完全独立 → 6 个 block 同时跑
threadIdx.x (0~1023)算 q·k 打分:线程 t 负责第 t、t+1024… 个历史位置块内线程分摊 pos+1 个时间步
__syncthreads等所有分数算完 → 块内 softmax → 再加权求 value共享内存 + 同步完成归约

⚠️ 注意这个朴素实现的关键特征:每个操作都是独立 kernel 启动(launch 一次、回来、再 launch 下一个)。这正是它好读的原因,也是它和你 ~/infer 项目的根本差异——后者把 RMSNorm+QKV+RoPE+KVCache 融合成一个 kernel,省掉中间的显存往返和启动开销,换取极致性能(但极难读)。
